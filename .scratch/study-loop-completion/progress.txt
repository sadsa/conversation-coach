## 03-srs-scheduling (2026-06-28)

Wired up FSRS scheduling via `ts-fsrs` on the PATCH handler and surfaced overdue chip on Vocabulary page.

**Key decisions:**
- `ts-fsrs` imported via dynamic `await import('ts-fsrs')` inside the PATCH handler (not at module top-level). Static import caused vitest's `vi.mock` hoisting to resolve and crash the ARM64 process before any tests ran; dynamic import defers the load to handler execution time where the ARM64 SIGILL only hits at process exit (expected).
- Pre-fetch pattern: PATCH now fetches the full item row first (session_id + all FSRS fields) in one query, then calls `verifyOwnedSession` on the session_id. Replaces the old `verifyOwnedViaSession` helper which did a separate select inside.
- `isRestudy` flag: `currentItem.reviewed && currentItem.stability != null` distinguishes first study (uses `createEmptyCard()`) from subsequent study (reconstructs Card from DB fields). `learning_steps: 0` is a safe default — not stored in DB, deprecated in ts-fsrs v6.
- `dueCount` computed in-memory in `loadPracticeItems` after mapping: `reviewed && due != null && new Date(due) <= now`. `due` typed as `due?: string | null` (optional) so existing test fixtures that omit it don't break.
- Vocabulary due chip uses `bg-violet-50 / text-violet-700` (dark: `bg-violet-900/20 / text-violet-300`) to match the violet accent theme; read-only `<p>` element (not a button).
- FSRS tests in separate file (`__tests__/api/practice-items-srs.test.ts`) to avoid module-cache interactions with `vi.resetModules()` calls in the original test file.

**Files changed:**
- `app/api/practice-items/[id]/route.ts` — pre-fetch + dynamic ts-fsrs import + FSRS write on reviewed: true
- `app/api/practice-items/route.ts` — destructure `{ items }` from new PracticeItemsResult return
- `app/vocabulary/page.tsx` — destructure `{ items, dueCount }`, pass dueCount to client
- `components/VocabularyClient.tsx` — dueCount prop + conditional chip

## 04-wild-capture (2026-06-28)

Implemented Wild Capture: "+" button on Vocabulary page → sheet → row created immediately → background Claude enrichment → flashcard fields populated.

**Key decisions:**
- Schema: `source TEXT NOT NULL DEFAULT 'annotation'` added, `session_id` made nullable, `user_id UUID` added to `practice_items` for direct ownership of manual items. CHECK constraint enforces: annotation items must have session_id, manual items must have user_id.
- Two-query loader: `loadPracticeItems` now fires two parallel queries — annotation items (via `!inner` session join) and manual items (via `user_id` direct). Combined and sorted; manual items always appear last.
- PATCH/DELETE ownership: added null-safe check — if `session_id` is null, verify via `user_id === user.id` directly. Both handlers fetch `user_id` in the initial select.
- Enrichment endpoint `POST /api/practice-items/[id]/enrich` — standalone endpoint called client-side after capture sheet closes. Calls `enrichWildCapture` (claude-haiku-4-5, structured JSON), updates flashcard fields, returns them.
- Client flow: POST creates row (returns id + phrase) → sheet closes → item added optimistically to state with shimmer → enrich called fire-and-forget → on resolve, item state updated with flashcard fields.
- `WildCaptureSheet`: DockedSheet form with textarea for phrase (data-initial-focus) and context. Footer submit button with spinner while submitting.
- `VocabularyList`: separates items by `source === 'manual' || session_id === null` into a "From real life" section rendered after all session groups.
- `WriteRow`: `enriching` prop shows phrase + "Enriching…" italic pulse instead of flashcard content. Falls back to StrikeOriginal once enrichment completes.
- Existing `makePracticeItemsDb` mock updated: `eqMock` now returns object with both `.order` and `.eq` to support the two-query parallel structure.
