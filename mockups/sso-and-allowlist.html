Mockups · SSO + self-serve allowlist

From email-the-owner to one-tap onboarding

Five surfaces compose the new sign-in flow: a Google-led login, a friendly pending-approval screen, an admin queue that fits on a phone, a lock-screen notification, and a single-click magic-link approval email. Each carries the same vocabulary already used elsewhere in the app — warm cream surfaces, violet for "pay attention", emerald reserved for voice, amber reserved for warnings.

01 · Login

Google first, magic-link as a fallback

Returning testers see a remembered identity and one-tap continue. New testers see the Google button above a quiet email form. The dark CTA is deliberate — it borrows the Practise Call door's "this is the loud one" treatment, but in primary navy rather than emerald (emerald stays reserved for voice surfaces).

02 · Pending approval

A patient waiting room, not a rejection

The screen a fresh signup lands on while the owner reviews their request. Amber for the icon (the "warning / paused" colour from the design system) rather than the red /access-denied palette — they haven't been rejected, just queued. The amber ring quietly pulses to suggest motion.

Your access request is in

I review new sign-ups personally — usually within a day. You'll get an email with a one-click sign-in link the moment you're approved. No need to come back here.

Requested as luciano.mateu@hotmail.com

Pending state. Single primary message, single secondary action. No mailto: button — there's nothing to email about; the next move is on the owner, not the user.

Why this isn't /access-denied

The existing /access-denied page stays put for the genuinely-denied case (red palette, "email me to ask why"). The pending screen is a distinct route with a distinct emotional register — amber, not red; "thanks, hold tight", not "you can't come in". Lumping them onto one page would force the copy to hedge.

Why pulse, why amber

A static screen reads as broken when waiting is the point. The oa-pulse keyframe family already establishes the ring-of- attention vocabulary; this borrows it gently. Amber is the only warning-adjacent hue the system allows; using it here is a small, honest cue that something is paused.

Copy choices

  • "I review" — first person owner voice, matches Access denied page.
  • "One-click sign-in link" — sets expectation that the email is the path back, not "try again later".
  • "No need to come back here" — explicit permission to close the tab.

03 · Admin

The approve action lives where your phone does

The owner will tap Approve from their phone after the lock-screen notification — so the admin surface is designed phone-first. Pending requests are the load-bearing content; approved and denied collapse into quiet collapsible sections beneath. Approve is the only primary action; Deny is a quiet outline button.

04 · Push notification

A nudge, not an alert

Reusing the existing VAPID push pipeline — your phone is already subscribed for "analysis ready" notifications, so adding "new access request" is a tiny extension, not a new dependency. Tapping the notification opens /admin with the new request at the top.

Lock-screen surface. Tone matches the existing "analysis ready" notification — same icon family, same brevity. App name kicker, one-line title, one-line context. Time stays "now" for fresh requests.

Notification copy formula

Title: always "New access request" — predictable, scannable in a stack of system notifications.
Body: "email signed in via provider. Tap to review."
URL: /admin — the service worker recognises this path and routes the tap there instead of /sessions/[id].

Why not email the owner?

The push path is already wired, tested, and reaches the owner's actual primary device. Adding Resend / Postmark for one type of transactional message would be premature. If push subscriptions later go per-user, the admin notification looks up the owner's endpoint explicitly rather than the single-row legacy pattern.

05 · Approval email

A door, not an announcement

When the owner taps Approve, the server calls auth.signInWithOtp({ email }). The user gets one email — a magic link that drops them straight into the app, no second login round. The body is short, warm, and on-brand: serif headline, body in Hanken, the same violet primary CTA used inside the app.

The "name fallback" branch

When the request came via Google we know the user's name — the headline reads "You're in, Luciano." When the request came via email magic-link we don't have a name yet; the headline falls back to "You're in." (no comma). Spec covers both branches.

Why one CTA, not two

The body could offer both "sign in now" and "sign in via Google later" — two paths. We don't. A single primary action lets the email be skim-readable and gives the user the fastest possible path back in. Adding choice here would slow the moment we're optimising for.

Sender + subject discipline

  • From: "Conversation Coach <noreply@…>" — matches the existing magic-link sender already in use.
  • Subject: "You're in — sign in to Conversation Coach". Direct, no exclamation marks, no emoji.
  • Body length: two short paragraphs above the fold; everything else can wait.
