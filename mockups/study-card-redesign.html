Study card redesign · concept 02

The Study queue, as a stack of bilingual phrase cards.

Today every Study row shows the same thing: the source sentence with the wrong fragment struck through and the rewrite inserted inline. That's a great way to read a correction; it's a noisier way to recall a phrase. The Study surface is downstream of Review — by the time the user lands here they have already chosen this item, they don't need to re-litigate the mistake. What they need is the corrected phrase in the language they're learning, anchored to its meaning in the language they already speak.

Two concept moves in this round:

1. Header alignment. Strip the subtitle and relocate the "Written archive" link so the list starts at the same vertical position as /review. Vertical cohesion across the methodology pillars.

2. Card content. Three card concepts using the existing flashcard_front / flashcard_back data Claude already produces — the native-language sentence above the target-language sentence (inverse for English learners). Each variant trades context density for recall focus differently.

Header alignment with Review

The list start position is the visual contract between the two pillars. The row chrome already matches — SessionList and WriteList both render rounded-xl cards on bg-surface with space-y-2 between rows — but Study's list still starts ~28px + 1.25rem lower than Review because of the calm one-liner subtitle and the ViewToggle row that hosts the "Written →" link. Drop both. The archive moves to a quiet pill below the list — retrospective destination, retrospective placement.

/review · reference

What the list start looks like today.

H1 → methodology eyebrow → space-y-8 → rounded session cards with the date pinned top-right. No subtitle, no filter row above the list. The page H1 is the only label; the list itself is the section. Study should land at the same vertical position with the same card vocabulary — SessionList and WriteList already share row chrome (rounded-xl, border-subtle, space-y-2), this concept finishes the cohesion at the list-start line.

Why the archive footer beats a top-of-list toggle:
  • The user comes to Study with a queue in mind, not an archive in mind. Top-right placement made the archive a peer of the queue; that's a hierarchy mismatch.
  • "Written" is a retrospective destination — the natural time to think about it is after you've scanned the queue. Footer placement matches the moment.
  • Only renders when writtenCount > 0. New users see clean alignment with Review; returning users with a real archive get a single quiet pill that anchors below the list.
  • The H1 row stays clean for a future bulk action / sort control if we ever need one.

Three card concepts

Each one uses flashcard_front and flashcard_back directly from Claude's existing output. The [[bracketed]] phrase is the focus — the surrounding sentence is just enough scaffolding to anchor it. The previous "strike-through-in-source-sentence" treatment moves to the sheet body, where the user has tapped in for the full story.

Concept A · Pair, target hero

Native prompt, target answer.

English sentence sits at the top in a quiet italic tertiary tone — it's the meaning, not the focus. The Spanish sentence drops into Source Serif 4 with the corrected phrase tinted and softly background-filled in the brand's correction green. The eye lands on the green phrase first, reads outward to anchor it in the sentence, then drifts up to confirm the meaning.

+ Closest to a real flashcard + Target phrase visually anchored ~ Loses the "I said this wrong" framing

Implementation checklist

New row component

  • Replace CorrectionInContext inside WriteRow with a new FlashcardRow (or pick the variant we commit to).
  • Source data: item.flashcard_front + item.flashcard_back already on PracticeItem; both contain a single [[bracketed]] phrase per the prompts in lib/claude.ts.
  • Parser: tiny helper that splits a flashcard string at the first [[…]] pair and returns { before, phrase, after }. Render in JSX so we never inject HTML.
  • Native vs target chooser: read targetLanguage from the user metadata header already in middleware.ts / getAuthenticatedUser. Pass into the row as a prop so the rule is centralised.
  • CorrectionInContext stays for the sheet body where the original-sentence framing still earns its space.

Header chrome (WriteClient + WriteList)

  • Drop <p>{t('write.subtitle')}</p> from WriteClient's header. Move the copy into the empty state if we want it preserved.
  • Remove the top <ViewToggle> from WriteList. Render archive link only as a footer pill (archive-footer in the mockup), gated on writtenCount > 0 && view === 'write'.
  • Written view itself keeps a tiny back-link near the top (the only way out of the archive), so the ViewToggle component shrinks but doesn't disappear — it just stops rendering on the primary Write view.
  • Result: list element starts at the same vertical position as the inbox on /review.

Fallbacks

  • Legacy practice items without flashcard_front / flashcard_back (analysis predates the field): fall back to the existing CorrectionInContext / StrikeOriginal treatment so old rows still render meaningfully.
  • Flashcard string missing brackets (defensive — Claude is supposed to always include them): render the whole sentence in the target weight without a tinted phrase. Don't blow up.
  • Empty state copy can absorb the calm "Pick up a saved correction whenever you're ready" subtitle if we want that line to live somewhere.

Out of scope

  • WriteSheet body — still uses the source-sentence correction-in-context. Sheet is for full story; the list is for recall.
  • Swipe-delete / undoable delete — unchanged.
  • Trailing fast-path "Mark as written" tap target — unchanged on the row chrome; new card variants all inherit it.
  • API / database — no schema change. Everything we need is already on practice_items.
  • Importance pill — not in the list today, not added by this round.

Open questions for the next pass

  • Concept A vs B vs C — which one earns the spot? A feels right for the patient-tutor tone; B helps with a long queue; C trades scan density for source attribution.
  • Should the source attribution from C migrate into A as a much quieter tertiary eyebrow above each row, or stay out of the list view entirely?
  • Is the green background tint on the corrected phrase too loud in a long list, or is the colour-as-anchor worth the visual weight?

Strings touched

  • write.subtitle — drops out of the populated header; can stay in the empty-state copy file under a renamed key (or be retired).
  • writeList.archiveLink — text moves to a centred footer pill ("N written · view archive →").
  • writeList.backToWrite — unchanged; still the top-of-Written-view back link.
