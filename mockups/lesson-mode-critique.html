Summary

What changes — and what gets deleted

  • DELETE · components/LessonPhaseRail.tsx + its test, the currentPhase state in LessonClient, the set_phase tool wiring on the client side. Keep the tool declared on the Gemini side so the model can still pace itself internally — its result no longer drives any UI.
  • DELETE · components/LessonPhrasePill.tsx stays — actually keep it as the only contextual cue. All four lesson.phase* i18n strings can be retired since nothing renders them.
  • P0 · adopt PracticeClient chrome — replace the lesson's bespoke top bar, listening-dot row, and lone mute button with the shared progress-rail + countdown + status-row + Mute/End-circle pattern. Mechanical lift from PracticeClient.tsx; same tokens, same animations. Files: components/LessonClient.tsx.
  • P0 · retry on analysis error — add Try again as primary in the 'error' branch (re-POSTs frozenTurnsRef.current); secondary is Discard. Mirrors PracticeClient's retry callback. Files: components/LessonClient.tsx.
  • P1 · connecting state — wrap (don't replace) <LoadingScreen /> with the lesson chrome: progress rail (empty) + phrase pill above, the existing robot loader below. Same component the rest of the app waits on; the lesson just adds its anchor. Continuous with the active state — no surface swap when Gemini reports setupComplete. Files: components/LessonClient.tsx.
  • Detail · Mute applies the same amber treatment + "Muted" status pill PracticeClient uses. Identical translation keys (practice.muteLabel / practice.unmuteLabel / practice.statusMuted) — drop the lesson-specific copies in i18n.ts.

Net effect: one component deleted, one component kept (LessonPhrasePill), LessonClient shortens by ~80 lines as the rail + bespoke control-row code disappears. Five fewer i18n strings. The lesson now reads as a third sibling to Call and Chat, distinguished only by the anchored phrase pill — which is the actual job-to-be-done difference.
